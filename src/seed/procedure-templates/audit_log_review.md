## 1. Purpose

This procedure establishes how audit logs generated by information systems at {{organisation_name}} are collected, protected, reviewed, and retained, enabling the detection of anomalous activity, security incidents, and policy violations.

## 2. Scope

{{scope}}

## 3. Log Sources and Events to Capture

| System Type | Minimum Events to Log |
|---|---|
| Authentication systems | Successful and failed logins, logouts, MFA events, account lockouts |
| Network (firewall, IDS/IPS) | Blocked connections, IDS/IPS alerts, VPN connections |
| Servers and endpoints | OS authentication, privilege escalation, service start/stop, software installation |
| Applications | User actions on sensitive data, admin operations, errors and exceptions |
| Cloud platforms | API calls, configuration changes, IAM changes, data access events |
| Security tools | Antivirus detections, vulnerability scan results, DLP alerts |
| This MCP server | All tool invocations, outcomes, and authentication events |

## 4. Roles and Responsibilities

- **{{owner}}**: Owns this procedure; accountable for log review programme
- **IT Security**: Operates SIEM and performs log reviews; escalates anomalies
- **IT Operations**: Ensures log collection agents and forwarding configurations are operational
- **System Owners**: Ensure their systems are generating and forwarding logs per this procedure
- **Internal Audit**: Reviews log review records and SIEM configuration during audits

## 5. Log Collection and Centralisation

1. All in-scope systems must forward logs to the centralised SIEM (Security Information and Event Management) system within 5 minutes of event generation.
2. Log forwarding is monitored for interruptions; gaps in log receipt trigger an alert within 1 hour.
3. Logs are transmitted over an encrypted channel (TLS 1.2+).
4. System clocks are synchronised to an authoritative time source (NTP) to ensure log timestamp accuracy.

## 6. Log Integrity and Protection

1. Logs must be write-protected: only authorised SIEM administrators may delete or modify logs.
2. Log tampering or deletion triggers an immediate security alert.
3. Logs are stored in a separate system from the systems that generated them to prevent local tampering.
4. Log access is restricted to authorised IT Security personnel and auditors; access is itself logged.

## 7. Log Review Schedule

| Review Type | Frequency | Performed By |
|---|---|---|
| Automated alerting (real-time) | Continuous | SIEM (automated) |
| Privileged account activity | Daily | IT Security |
| Authentication anomalies | Daily | IT Security |
| Security control alerts | Daily | IT Security |
| Full log review (trend analysis) | Weekly | IT Security |
| Compliance-focused review | Monthly | IT Security + {{owner}} |

## 8. Review Process

### 8.1 Daily Automated Alert Triage
1. IT Security reviews all SIEM alerts generated in the previous 24 hours at the start of each business day.
2. Each alert is assessed and classified as: True Positive, False Positive, or Requires Further Investigation.
3. True Positives that indicate a security incident are escalated via the Incident Handling Procedure.
4. False Positives are documented and, where recurring, used to tune SIEM rules.
5. Alert triage is recorded in the SIEM or ticketing system.

### 8.2 Weekly Log Review
1. IT Security performs a structured review of logs from the previous 7 days, focusing on:
   - Unusual login patterns (time of day, geographic location, volume)
   - Privilege escalation events
   - Bulk data access or export events
   - Configuration changes (firewall rules, IAM policies, security tool settings)
   - Failed authentication trends
2. Findings are recorded in the weekly review log.
3. Anomalies not already captured by automated alerting are investigated and, if confirmed as incidents, escalated via the Incident Handling Procedure.

### 8.3 Monthly Compliance Review
1. {{owner}} and IT Security conduct a monthly review covering:
   - Audit log completeness (all in-scope systems forwarding logs)
   - SIEM health (alert rules up to date, no suppressed critical rules)
   - Compliance with log retention requirements
   - Outstanding findings from previous reviews
2. A monthly review report is produced and approved by {{owner}}.

## 9. Anomaly Indicators

IT Security must investigate any of the following when detected:
- Login attempts from unusual geographic locations or IP ranges not previously seen
- Account activity outside normal business hours without prior notice
- Multiple failed authentication attempts followed by a success (credential stuffing indicator)
- Bulk access or download of records (potential data exfiltration)
- Administrative actions performed outside of change management windows
- Disabled or reconfigured logging on any in-scope system
- Privilege escalation not linked to an approved access request

## 10. Log Retention

Audit logs are retained in accordance with the following minimum periods:
- Security logs (authentication, network, SIEM alerts): 12 months online + 2 years archived
- Application logs: 12 months
- Compliance-critical logs (financial systems, personal data access): 5 years

Retention periods may be extended by legal holds or regulatory requirements.

## 11. Record Keeping

Daily triage records, weekly review logs, monthly review reports, and SIEM configuration change records are retained for a minimum of 3 years.
